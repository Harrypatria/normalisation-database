What is Normalization?

A systematic approach to organizing data in a database

Core Objectives

  • Minimize redundancy — Store data once
  • Eliminate anomalies — Prevent data corruption
  • Ensure integrity — Maintain consistency
  • Optimize performance — Faster operations

Key Benefits

  • No duplicate data storage
  • Prevention of update errors
  • Automatic consistency checks
  • Improved query efficiency

Key Principle: Organize data to reduce redundancy while maintaining all necessary relationships

Foundation: E.F. Codd (1970)

Essential Concepts

Primary Key (PK)

Definition: Unique identifier for each record in a table

StudentID → 12345 Uniquely identifies each student

Must be unique and non-null

Foreign Key (FK)

Definition: Field referencing a PK in another table

StudentID in Enrollments → References Students.StudentID

Creates relationships between tables

Functional Dependency

Definition: Attribute A determines attribute B

StudentID → Name If you know StudentID, you know Name

Notation: A → B

Critical: Understanding these concepts is essential for proper normalization

The Normal Forms Hierarchy

Progressive Refinement of Data Structure

UNF Unnormalized — Multi-valued attributes
1NF First Normal Form — Atomic values only
2NF Second Normal Form — No partial dependencies
3NF Third Normal Form — No transitive dependencies

Working Example: Student Course Enrollment System — Each level eliminates specific data problems

First Normal Form (1NF): Atomic Values

Rule: Each column must contain atomic (indivisible) values — no lists or sets

❌ Before 1NF

StudentID Name Courses
1 Alice Math, Physics
2 Bob Chemistry, Math

Problems:

  • Non-atomic "Courses" column
  • Cannot query individual courses
  • Difficult to add/remove courses

✅ After 1NF

StudentID Name Course
1 Alice Math
1 Alice Physics
2 Bob Chemistry

Benefits:

  • All values are atomic
  • Easy to query specific courses
  • Simple to maintain
CREATE TABLE student_courses ( student_id INTEGER, name TEXT, course TEXT, PRIMARY KEY (student_id, course) );

Second Normal Form (2NF): No Partial Dependencies

Rule: Must be in 1NF + All non-key attributes depend on the entire primary key

❌ Before 2NF — The Problem

StudentID Course Name Grade
1 Math Alice A
1 Physics Alice B
2 Math Bob B

Primary Key: (StudentID, Course)

Issue: Name depends only on StudentID (partial dependency) → Data redundancy

✅ After 2NF — Decomposed Tables

Students Table

StudentIDName
1Alice
2Bob

Enrollments Table

StudentIDCourseGrade
1MathA
1PhysicsB
2MathB
CREATE TABLE students ( student_id INTEGER PRIMARY KEY, name TEXT ); CREATE TABLE enrollments ( student_id INTEGER, course TEXT, grade TEXT, PRIMARY KEY (student_id, course), FOREIGN KEY (student_id) REFERENCES students(student_id) );

Third Normal Form (3NF): No Transitive Dependencies

Rule: Must be in 2NF + No transitive dependencies (non-key → non-key)

❌ Before 3NF — Transitive Dependency

StudentID Course Instructor Department
1 Math Dr. Smith Science
1 Physics Dr. Jones Science

Problem: Department depends on Instructor (not on the key) → Instructor → Department

✅ After 3NF — Optimal Structure

Enrollments Table

StudentIDCourseInstructor
1MathDr. Smith
1PhysicsDr. Jones

Instructors Table

InstructorDepartment
Dr. SmithScience
Dr. JonesScience

Benefits: No transitive dependencies | Minimal redundancy | Easy updates

CREATE TABLE instructors ( instructor TEXT PRIMARY KEY, department TEXT ); CREATE TABLE enrollments ( student_id INTEGER, course TEXT, instructor TEXT, PRIMARY KEY (student_id, course), FOREIGN KEY (instructor) REFERENCES instructors(instructor) );

Hands-On Exercise: Library System

Your Challenge

Normalize the following library database to 3NF. Identify problems and design the optimal schema.

Current (Unnormalized) Design

BookID Title Authors BorrowerID BorrowerName
101 Database Systems Smith, Jones 1001 Alice
102 AI Fundamentals Brown 1002 Bob
103 Web Development Smith, Chen 1001 Alice

Questions to Answer

  1. What normal form is this table currently in?
  2. What are the functional dependencies?
  3. What anomalies exist?
  4. How many tables do you need for 3NF?

Hints

  • Authors column has multiple values
  • BorrowerName depends on BorrowerID
  • Think about insert/update/delete issues
  • Apply 1NF → 2NF → 3NF systematically

Work individually or in pairs | 3 minutes | Solution on next slide

Solution: Library System in 3NF

✅ Normalized Schema (5 Tables)

Tables & Keys:
  • Books: BookID (PK), Title
  • Authors: AuthorID (PK), AuthorName
  • Book_Authors: BookID (FK), AuthorID (FK)
  • Borrowers: BorrowerID (PK), Name
  • Loans: LoanID (PK), BookID (FK), BorrowerID (FK)
Transformation Steps:
  1. 1NF: Split multi-valued Authors
  2. 2NF: Separate Books & Authors entities
  3. 3NF: Extract Borrowers (BorrowerID → Name)
  4. Result: Junction table for many-to-many
-- Core Entity Tables CREATE TABLE books (book_id INTEGER PRIMARY KEY, title TEXT); CREATE TABLE authors (author_id INTEGER PRIMARY KEY, name TEXT); CREATE TABLE borrowers (borrower_id INTEGER PRIMARY KEY, name TEXT); -- Relationship Tables CREATE TABLE book_authors ( book_id INTEGER, author_id INTEGER, FOREIGN KEY (book_id) REFERENCES books(book_id), FOREIGN KEY (author_id) REFERENCES authors(author_id) ); CREATE TABLE loans ( loan_id INTEGER PRIMARY KEY, book_id INTEGER, borrower_id INTEGER, FOREIGN KEY (book_id) REFERENCES books(book_id), FOREIGN KEY (borrower_id) REFERENCES borrowers(borrower_id) );

Achievement: Complete elimination of redundancy and anomalies | Flexible & scalable design

Summary & Professional Best Practices

Key Takeaways

  • Normalization solves real problems: 85% redundancy reduction, $15M+ savings
  • Three normal forms: 1NF (atomic) → 2NF (no partial deps) → 3NF (no transitive deps)
  • Functional dependencies are the foundation of proper design
  • Data integrity through constraints and relationships

When to Normalize

  • Transactional systems (OLTP)
  • Frequently updated data
  • Multi-user environments
  • Systems requiring high integrity

Common Mistakes

  • Over-normalization beyond 3NF
  • Ignoring functional dependencies
  • Missing foreign key constraints
  • Poor documentation
